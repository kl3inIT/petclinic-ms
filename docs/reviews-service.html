reviews-service

Service quản lý đánh giá đa hình (polymorphic review) — cùng schema dùng cho VET, PRODUCT, và VISIT. Tài liệu sketch nghiệp vụ trước khi vào phase plan/execute.

1 aggregate Review với (targetType, targetId) chỉ xác định đối tượng được review. Customer chỉ được review khi đã tương tác hợp lệ (visit COMPLETED hoặc bill PAID). Review qua moderation pipeline (PENDING_MODERATION → PUBLISHED/HIDDEN). Khi summary thay đổi, fan-out event review.summary.updated để vets/products cập nhật averageRating denormalized.

1. Personas & vai trò

👤 Customer (USER)

Quyền: tạo / sửa (≤7 ngày) / xóa mềm review của mình; vote helpful; xem mọi review PUBLISHED.

Điều kiện tạo: phải có visit COMPLETED (cho VET/VISIT) hoặc bill PAID chứa product (cho PRODUCT).

🛡️ Staff (RECEPTIONIST / VET)

Quyền: xem moderation queue, approve / hide review (kèm lý do); không xóa cứng.

Không thể: tạo review hộ customer.

👑 Admin

Quyền: mọi quyền của Staff + xóa cứng (hard delete) review vi phạm pháp lý.

Audit: mọi action ghi vào review_audit_log.

2. Bounded context

Sở hữuKhông sở hữu
  • Review aggregate (entity + lifecycle)
  • Eligibility rule engine
  • Moderation pipeline (queue + actions)
  • Summary tính toán (avg, count, distribution)
  • Helpful vote
  • Review audit log
  • Vet profile (vets-service sở hữu) — chỉ giữ targetId
  • Product catalog (products-service sở hữu) — chỉ giữ targetId
  • Visit entity (visits-service sở hữu) — chỉ giữ targetId
  • User identity (auth-service sở hữu) — chỉ snapshot authorName
  • Bill (billing-service sở hữu) — chỉ query để check eligibility
Quy tắc vàng: review-service KHÔNG có foreign key xuyên service. Mọi targetId / authorId chỉ là UUID lưu trữ. Validation xảy ra qua HTTP call (eligibility) hoặc qua event (snapshot tên author).

3. ERD — Review aggregate

erDiagram REVIEW ||--o{ REVIEW_VOTE : "có nhiều" REVIEW ||--o{ REVIEW_AUDIT_LOG : "có nhiều" REVIEW { uuid id PK string target_type "ENUM VET PRODUCT VISIT" uuid target_id uuid author_id "user.id" string author_name "snapshot" int rating "1..5" string title "nullable, max 120" text comment "nullable, max 2000" string status "ENUM PUBLISHED PENDING_MODERATION FLAGGED HIDDEN DELETED" string moderation_reason "nullable" int helpful_count timestamp created_date timestamp last_modified_date string created_by string last_modified_by } REVIEW_VOTE { uuid id PK uuid review_id FK uuid voter_id "user.id" bool is_helpful timestamp voted_at } REVIEW_AUDIT_LOG { uuid id PK uuid review_id FK string action "ENUM CREATED EDITED APPROVED HIDDEN DELETED FLAGGED" string actor_id string actor_role text detail "JSON delta" timestamp created_date } PROCESSED_EVENT { uuid event_id PK string event_type timestamp processed_at }

Index & ràng buộc

  • UNIQUE (author_id, target_type, target_id) — 1 user chỉ review 1 đối tượng 1 lần
  • UNIQUE (review_id, voter_id) trên review_vote
  • INDEX (target_type, target_id, status) — query chính: list review theo target
  • INDEX (status, created_date) — moderation queue sort theo cũ nhất
  • processed_event — idempotency table cho consumer (gotcha #20)

4. State machine — Review status

stateDiagram-v2 [*] --> PENDING_MODERATION : POST /reviews (nếu auto-moderation FAIL hoặc tắt) [*] --> PUBLISHED : POST /reviews (auto-moderation PASS) PENDING_MODERATION --> PUBLISHED : staff approve PENDING_MODERATION --> HIDDEN : staff hide (reason) PUBLISHED --> FLAGGED : user report (>= N báo cáo) PUBLISHED --> HIDDEN : staff hide (reason) PUBLISHED --> DELETED : author soft delete (trong 7 ngày) FLAGGED --> PUBLISHED : staff approve sau review FLAGGED --> HIDDEN : staff hide HIDDEN --> [*] : admin hard delete DELETED --> [*] : admin hard delete
Quy tắc edit: author chỉ được PATCH trong 7 ngày kể từ created_date. Edit reset status về PENDING_MODERATION nếu nội dung thay đổi đáng kể (rating ±2 sao hoặc comment thay >30% ký tự).

7. Sequence 3 — Summary fan-out (sau khi review thay đổi)

sequenceDiagram autonumber participant R as reviews-service participant MQ as RabbitMQ topic petclinic.events participant V as vets-service participant P as products-service participant ML as mailer-service (Go) Note over R: Trigger: review created / approved / hidden / deleted R->>MQ: publish review.summary.updated.vet
{eventId, targetType:VET, targetId, avgRating, count} R->>MQ: publish review.created
{eventId, targetType, targetId, authorId, rating} par Fan-out song song MQ->>V: queue vets.review.summary.updated.vet V->>V: Dedupe by eventId
(processed_events table) V->>V: UPDATE vet SET average_rating=?, review_count=? and MQ->>P: queue products.review.summary.updated.product
(chỉ khi targetType=PRODUCT) P->>P: UPDATE product SET average_rating=? and MQ->>ML: queue mailer.review.created ML->>ML: Compose email "Cảm ơn đánh giá" ML->>ML: SMTP → Mailpit (dev) / SES (prod) end Note over V,P: vets/products giữ averageRating denormalized
→ list page không phải call reviews-service N lần
Vì sao fan-out qua event thay vì HTTP: (1) reviews-service không cần biết ai consume — thêm consumer mới (vd. analytics-service) không phải sửa publisher. (2) Decouple uptime: vets-service down không chặn customer tạo review. (3) Tránh N+1 khi vets list page render — đọc average_rating local thay vì call sang reviews-service N lần.

11. Câu hỏi mở (cần align với team trước khi /gsd-plan-phase)

  1. Phase 1 scope: Bắt đầu với 2 target type (VET + VISIT) — defer PRODUCT đến khi billing-service ra? Hay làm cả 3 nhưng PRODUCT eligibility tạm bypass?
  2. Auto-moderation engine: Phase 1 dùng regex profanity filter (Vietnamese + English wordlist)? Hay tích hợp OpenAI Moderation API ngay? Trade-off: regex free + offline; OpenAI tốt hơn nhưng cần API key + chi phí.
  3. Edit window: 7 ngày có quá ngắn/dài? Champlain reference không có rule này — chốt với giảng viên.
  4. Visit review vs Vet review: 1 visit chỉ 1 review nhưng 1 vet nhiều review. UI có nên hỏi "Bạn muốn review BÁC SĨ hay BUỔI KHÁM?" khi customer vào trang visit hoàn thành?
  5. Báo cáo (report) threshold: ≥ N báo cáo → auto FLAGGED. Chốt N = 3 hay 5?
  6. Anonymous review: Có cho phép review ẩn danh (giấu authorName)? Champlain không có — khả năng cao là KHÔNG (compliance + trách nhiệm).
  7. Port: 818X — chốt port (đề xuất 8186 vì 8185 là admin-server, 8187 là mcp-server).
  8. vets-service averageRating field: Champlain reference đã có Rating bên trong vets-service. Khi port qua petclinic-ms phải bỏ entity Rating khỏi vets-service và chỉ giữ field denormalized averageRating: Double. Cần Dev vets-service align.
