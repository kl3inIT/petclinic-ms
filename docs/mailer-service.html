Mailer Service Go

Async email worker — consumer-only. Subscribe RabbitMQ → render template → SMTP. Một trong tổng số ~8 service của petclinic-ms, viết bằng Go vì pure I/O không cần JPA hay OAuth.

Mailer consume petclinic.events exchange → render HTML template → gửi qua SMTP. Idempotency dedupe qua Redis. DLQ tự động cho message lỗi. Một dịch vụ duy nhất viết Go thay vì Spring, lý do: stateless, không transaction, không OAuth, footprint nhỏ (25 MB so với 250 MB của JVM).

Tại sao Go (không Spring)?

Mailer là consumer thuần — không có business logic phức tạp, không JPA, không cross-service HTTP call ngược, không OAuth issuer. Workload của Go: I/O nhiều, RAM ít, restart nhanh.

Nguyên tắc chọn ngôn ngữ trong petclinic-ms

  • Spring cho service có JPA + transaction + Security stack: auth, customers, vets, visits, billing, gateway
  • Go cho service stateless pure-I/O: mailer (giờ), tương lai có thể là file-service (upload + resize ảnh thú cưng), notification (WebSocket realtime)
  • Không đa ngôn ngữ vì sở thích — chỉ khi có lý do thực tế (footprint, concurrency, library ecosystem)

Tại sao publish AFTER save mà không trong transaction?

Theo lý thuyết, transactional outbox là pattern chuẩn (lưu event vào bảng outbox cùng transaction với save user, một job riêng đọc outbox → publish). Project này chưa làm outbox vì:

  • Welcome mail mất 1 email không nguy hiểm (idempotent từ góc người dùng — register lại được)
  • Outbox thêm bảng + job + complexity — không xứng đáng cho v1
  • Code đã try/catch — broker down chỉ log warning, register vẫn ok

Khi đến event invoice.created hoặc payment.received (tiền thật), outbox sẽ bắt buộc — sẽ thêm sau.

Tolerant Reader pattern

Go service không thể import Java module common-events. Pattern: redefine struct với JSON tag, chỉ field consumer cần. Jackson bên publisher serialize full record, Go side unmarshal sẽ ignore field thừa.

Publisher (Java)

public record UserRegisteredEvent(
    UUID eventId,
    String eventType,
    Instant occurredAt,
    String source,
    UUID userId,
    String username,
    String email
) implements DomainEvent { ... }

Consumer (Go)

type UserRegistered struct {
    EventID    string    `json:"eventId"`
    EventType  string    `json:"eventType"`
    OccurredAt time.Time `json:"occurredAt"`
    Source     string    `json:"source"`

    UserID   string `json:"userId"`
    Username string `json:"username"`
    Email    string `json:"email"`
}

Publisher sau thêm field mới (vd locale) — Go side k cần update gì, vẫn parse OK. Đây là điểm mạnh của loose coupling qua schema JSON.

Idempotency: SETNX với TTL

Một message AMQP có thể được redeliver nhiều lần (broker restart, consumer crash giữa ack, network jitter). Mailer cần dedupe để khỏi gửi mail 2 lần.

internal/store/redis.go
func (i *Idempotency) Claim(ctx context.Context, eventID string) (bool, error) {
    key := "mailer:processed:" + eventID
    ok, err := i.client.SetNX(ctx, key, "1", i.ttl).Result()
    // ok=true → key chưa tồn tại, claim thành công, được gửi mail
    // ok=false → key đã có, ai đó claim rồi, skip
    return ok, err
}
ScenarioBehavior
Lần đầu nhận eventSETNX OK → send → ack
Redeliver sau ack thành công nhưng RMQ chưa nhậnSETNX returns 0 → skip → ack (no double mail)
Crash sau SETNX, trước SendEdge case Mất mail. Acceptable cho welcome (user register lại được).
Sau 7 ngày (TTL)Key tự xoá. Cùng eventId sau 7 ngày sẽ resend — không xảy ra trong thực tế.

Production thật cần exactly-once?

Cho mail tiền (invoice, payment receipt), pattern này chưa đủ. Cần:

  • Outbox phía publisher → đảm bảo event lên broker
  • Idempotency key trong table phía consumer (không phải Redis — Redis có thể flush) cùng transaction với side effect (vd: insert vào mail_sent table khi send xong, FK với processed_event_id)
  • SMTP provider có idempotency header (SendGrid, SES có)

Hiện tại scope là welcome mail → SETNX + TTL đủ.

DLQ + retry

Khi handler trả lỗi (vd SMTP timeout, template render fail, payload corrupt) → Nack(requeue=false) → message rớt DLQ qua x-dead-letter-exchange.

internal/consumer/consumer.go
func (c *Consumer) dispatch(ctx context.Context, d amqp.Delivery, handler Handler) {
    if err := handler(ctx, d.Body); err != nil {
        c.log.Error("handler failed -> DLQ", "err", err, "routingKey", d.RoutingKey)
        _ = d.Nack(false, false)   // requeue=false → DLQ
        return
    }
    _ = d.Ack(false)
}

Tại sao không retry trong process? Vì Go service stateless — retry cùng message khi lỗi transient (SMTP 4xx) sẽ làm consumer chậm. Tốt hơn: đẩy DLQ → 1 job riêng (chưa làm) replay DLQ vào main queue sau X phút (exponential backoff).

Hiện tại operator inspect DLQ bằng tay tại RabbitMQ UI → Queues → click mailer.user.registered.dlq → "Get message" để xem payload + lỗi.

Graceful shutdown

Go signal handling đơn giản hơn Spring nhiều — không có @PreDestroy, chỉ có signal.NotifyContext + context.Cancel.

cmd/mailer/main.go
ctx, cancel := signal.NotifyContext(context.Background(), os.Interrupt, syscall.SIGTERM)
defer cancel()

// ... start consumer in goroutine, pass ctx ...

<-ctx.Done()                          // wait for SIGINT/SIGTERM
shutdownCtx, _ := context.WithTimeout(context.Background(), 10*time.Second)
_ = srv.Shutdown(shutdownCtx)         // HTTP graceful shutdown
wg.Wait()                             // wait consumer goroutine return

Trong consumer: select { case <-ctx.Done(): return } — không poll, không sleep. Đây là Go idiom standard.

Cách chạy

1. Start infrastructure (chỉ 1 lần)

docker compose --profile mq --profile cache --profile mail up -d

Khởi động RabbitMQ + Redis + Mailpit. Xem RabbitMQ UI tại localhost:15672 (guest/guest), Mailpit tại localhost:8025.

2. Start auth-service (publisher)

./gradlew :services:auth-service:bootRun

Hoặc chạy compound 🐱 Petclinic Apps từ IntelliJ.

3. Start mailer-service

cd services/mailer-service
go run ./cmd/mailer

Log JSON xuất hiện báo "amqp connected", "templates loaded", "subscribed".

4. Trigger event

curl -X POST http://localhost:8180/api/v1/auth/register \
  -H "Content-Type: application/json" \
  -d '{"username":"test","email":"test@example.com","password":"P@ssw0rd!"}'

Mở Mailpit UI — phải thấy mail mới với subject "Chào mừng đến Petclinic MSS301".
