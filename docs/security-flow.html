1. Tổng quan

Auth-service phát hành JWT. Mỗi service downstream tự validate JWT (zero-trust). Gateway là defense-in-depth lớp ngoài. Bắt đầu HMAC simple, tiến tới RSA + JWKS production-grade.

Roadmap — 4 iterations (OAuth/MFA defer)

IterScopeCryptoTrust model
Iter 1 auth-service alone — register/login/me HS256 (HMAC, shared secret) Self only
Iter 2 shared/common-security + wire gateway + customers/vets validate HS256 (secret via config-repo) Defense-in-depth (gateway + each service)
Iter 3 Switch sang RSA + JWKS endpoint + refresh token flow RS256 (RSA keypair, JWKS rotation) Production-grade asymmetric
Iter 4 Service-to-service token forward + gateway rate limit + audit log structured (ECS) Same (RS256) Full zero-trust + observability
Defer OAuth2 social login (Google/GitHub), MFA TOTP, password reset — —

3. JWT structure

JWT = 3 phần base64url-encoded, ngăn cách bằng dấu chấm:

eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9
.
eyJpc3MiOiJwZXRjbGluaWMtbXMiLCJzdWIiOiI3YjMyZjAxLWMwMTQtNDM3NS04YjY1LTQ1MDhhNTE4MzczMyIsInJvbGVzIjpbIlVTRVIiXSwiaWF0IjoxNzMwNzkyMDAwLCJleHAiOjE3MzA3OTI5MDB9
.
7XAJpQs3vP6h5C8nN4kMz_xK2hYwGfL9aB...

Header (red) — base64decode

{
  "alg": "HS256",     // HMAC SHA-256 (Iter 1-2). Iter 3 đổi sang RS256
  "typ": "JWT"
}

Payload (purple) — claims

{
  "iss": "petclinic-ms",                          // issuer
  "sub": "7b32f01-c014-4375-8b65-4508a5183733",   // subject = user.id (UUID)
  "aud": "petclinic-ms",                          // audience — services check this
  "iat": 1730792000,                              // issued at
  "exp": 1730792900,                              // expires (15 min later)
  "username": "alice",                            // custom claim
  "roles": ["USER", "VET"]                        // RBAC
}

Signature (blue)

HMACSHA256(
  base64UrlEncode(header) + "." + base64UrlEncode(payload),
  SECRET  // 256-bit secret từ config-repo (Iter 1-2)
)
Quan trọng: Payload KHÔNG được mã hóa — ai cũng decode được. Signature CHỈ chứng minh token chưa bị sửa đổi + được issuer cấp. KHÔNG bao giờ đặt password/secret trong claims.

4. Iteration 2 — defense-in-depth Iter 2

Wire JWT validation ở HAI lớp: gateway (đẩy request hỏng sớm) + mỗi service downstream (zero-trust — không tin internal network).

:8080 participant CUST as customers-service
:8181 U->>GW: GET /api/v1/owners
Authorization: Bearer eyJhbGciOi... rect rgb(22, 78, 99) Note over GW: Defense lớp 1 — Gateway GW->>GW: SecurityFilterChain decode JWT
(common-security AutoConfig) alt token hợp lệ GW->>GW: extract sub, roles vào MDC
(audit log) else token hỏng GW-->>U: 401 (kill sớm, KHÔNG gọi downstream) end end GW->>CUST: forward request
Authorization header KÉO theo
+ X-Request-Id correlation rect rgb(120, 53, 15) Note over CUST: Defense lớp 2 — Service CUST->>CUST: SecurityFilterChain decode JWT lần nữa
(common-security AutoConfig — SAME module) alt token vẫn hợp lệ CUST->>CUST: @PreAuthorize("hasRole('USER')") check CUST-->>GW: 200 + data GW-->>U: 200 + data else mismatch / fake CUST-->>GW: 401 GW-->>U: 401 end end

Tại sao validate 2 lần?

Defense-in-depth nguyên tắc: Không bao giờ giả định inbound traffic đã được auth. Attacker có thể:
  • Bypass gateway nếu khám phá ra service IP trực tiếp (k8s pod IP, security group sai)
  • Compromise gateway service → tự forward malicious header xuống downstream
  • Service mesh proxy bug → header bị inject sai
Cost: ~1-3ms overhead per service per request (HMAC verify nhanh).

Mỗi service config Security giống nhau → tách vào shared module, auto-wire:

5. Iteration 3 — RSA + JWKS + refresh tokens Iter 3

5.1 Asymmetric crypto

Vấn đề HMAC ở Iter 1-2: cùng 1 secret distribute mọi service. Nếu customers-service bị compromise → secret leak → attacker forge JWT giả mạo bất kỳ user.

RSA fix:
  • auth-service: giữ private key (sign tokens)
  • customers/vets/gateway: lấy public key qua JWKS endpoint → chỉ verify, không thể sign
  • Service compromise → chỉ public key leak (không nguy hại)
  • Key rotation: auth-service rotate private, JWKS endpoint publish new + old, clients pick up automatically

5.2 JWKS endpoint pattern

sequenceDiagram participant CUST as customers-service participant AUTH as auth-service Note over CUST: Startup hoặc cache miss CUST->>AUTH: GET /.well-known/jwks.json AUTH-->>CUST: {
"keys": [
{kid:"k1", kty:"RSA", n:"...", e:"AQAB", alg:"RS256"}
]
} CUST->>CUST: Cache 24h (NimbusJwtDecoder.withJwkSetUri) Note over CUST: Request validation CUST->>CUST: Parse JWT header → kid="k1" CUST->>CUST: Lookup k1 in cached JWKS
verify signature Note over AUTH: Key rotation (every 90 days) AUTH->>AUTH: Generate new keypair "k2" AUTH->>AUTH: JWKS now serves [k1, k2]
Sign new tokens with k2 Note over CUST: Cache miss for k2 CUST->>AUTH: GET /.well-known/jwks.json (auto refresh) AUTH-->>CUST: [k1, k2] CUST->>CUST: Old tokens (kid=k1) vẫn validate
New tokens (kid=k2) cũng validate Note over AUTH: After all k1 tokens expired (15min later) AUTH->>AUTH: JWKS drops k1 → [k2]

5.3 Refresh token flow

Access token cố tình ngắn (15min) để giảm impact nếu leak. User không muốn login lại mỗi 15 phút → refresh token (7-30 ngày) đổi access token mới.
sequenceDiagram autonumber actor U as User participant A as auth-service participant D as Postgres Note over U,A: Initial login U->>A: POST /login {username, password} A->>D: INSERT refresh_tokens
{id=UUID-v7, user_id, hashed_token, expires_at=now+7d, revoked=false} A-->>U: {accessToken (15min), refreshToken (opaque UUID), expiresIn} Note over U,A: After 15min, access token expired U->>A: POST /api/v1/auth/refresh
{refreshToken: "..."} A->>D: SELECT WHERE hash=? AND revoked=false AND expires_at>now alt valid A->>D: UPDATE SET revoked=true (rotate — single-use) A->>D: INSERT new refresh_token A-->>U: {accessToken (new), refreshToken (new)} else revoked or expired A-->>U: 401 → user phải login lại end Note over U,A: Logout U->>A: POST /logout (with refresh token) A->>D: UPDATE refresh_tokens SET revoked=true WHERE user_id=?

6. Iteration 4 — service-to-service + rate limit + audit Iter 4

6.1 Token forwarding (service A calls service B internal)

Khi visits-service cần data từ customers-service:

sequenceDiagram actor U as User participant GW as gateway participant VIS as visits-service participant CUST as customers-service U->>GW: GET /api/v1/visits/123
Authorization: Bearer USER_JWT GW->>VIS: forward (Auth header passthrough) VIS->>VIS: validate USER_JWT — OK Note over VIS,CUST: Internal call — cần auth! VIS->>CUST: GET /api/v1/owners/456
Authorization: Bearer USER_JWT
(propagate user's token, NOT a service token) CUST->>CUST: validate USER_JWT
check @PreAuthorize("hasRole('USER')") CUST-->>VIS: owner data VIS-->>GW: enriched visit + owner GW-->>U: 200
Spring Security 7: RestClient + OAuth2ResourceServerHttpAdvice tự forward token. Hoặc dùng ClientHttpRequestInterceptor lấy từ SecurityContext, gắn vào outbound request.

  "event.action": "authentication.login.success",

  "client.ip": "203.0.113.10",
  "user_agent.original": "Mozilla/5.0...",

Alert rule: nếu event.action=authentication.login.failed > 5 trong 1 phút cùng IP → page on-call.

8. RBAC model

Roles tĩnh, lưu CSV trong cột users.roles_csv (đơn giản — đủ cho học, scale lên Role entity riêng sau khi cần dynamic roles):

USER VET ADMIN
RoleQuyền

}
