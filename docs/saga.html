1. Tại sao cần Saga?

Vấn đề: Distributed transaction không thể dùng ACID 2-Phase Commit

Mỗi microservice có DB riêng. Khi 1 business flow đụng nhiều service (booking → payment → inventory → email), nếu 1 bước fail giữa chừng, bạn KHÔNG thể "ROLLBACK" như SQL monolith. 2PC tồn tại nhưng chậm, locking heavy, single point of failure.

Saga: chuỗi local transactions với compensating actions

Mỗi service làm local TX trong DB của nó. Nếu bước cuối fail → các bước trước thực thi compensating transaction (undo business level, không phải SQL rollback).

Ví dụ: refund payment thay vì xóa row payment. Cancel inventory hold thay vì DELETE inventory_reservation. Mail apology thay vì un-send mail.

2. Khái niệm cốt lõi

Khái niệmÝ nghĩa
Local transactionĐơn vị work atomic trong 1 service / 1 DB. Saga step = local TX.
Compensating transactionBusiness action undo local TX trước. KHÔNG phải SQL rollback — vì TX đã commit + có thể trigger side effects (event publish).
Saga statePENDING → COMPLETED (happy path) hoặc PENDING → COMPENSATED (failure path). Lưu phân tán ở initiator service.
IdempotencyMỗi step + compensation phải an toàn khi retry (broker at-least-once). Dedupe theo eventId.
ChoreographyKhông có orchestrator. Service tự listen event của nhau → trigger step kế.
OrchestrationCó 1 saga manager central. Gọi từng service, nhận response, quyết định step kế hoặc compensate.

3. Choreography vs Orchestration

Choreography petclinic-ms dùng cái này

  • Decentralized — mỗi service publish event, các service khác listen.
  • Đơn giản với flow ≤ 4-5 steps.
  • Khó debug khi flow phình ra — không nhìn được "đang ở step nào" trong 1 chỗ.
  • Coupling event-based — service phải biết event name + schema của nhau.
  • Tools: Spring AMQP + RabbitMQ (như petclinic), Kafka.

Orchestration prod-grade cho flow lớn

  • Centralized — 1 SagaManager workflow drive luồng.
  • Visibility cao — query "saga X đang ở step nào?" 1 truy vấn.
  • Coupling tới orchestrator — service chỉ là worker, không tự quyết.
  • Failure handling cleaner — orchestrator biết exact step nào fail → compensate ngược order.
  • Tools 2026: Temporal, Eventuate Tram, Camunda, AWS Step Functions.
Industry rule of thumb (2026): Choreography cho ≤ 5 steps + decoupled domain. Orchestration cho mọi flow lớn hơn hoặc khi cần observability cao (financial / compliance).

4. Flow trong petclinic-ms

4.1. Happy path — mail gửi OK

sequenceDiagram autonumber participant V as visits-service participant DB as visits.DB participant MQ as RabbitMQ participant M as mailer-service (Go) participant SMTP as Mailpit (SMTP) V->>DB: BEGIN TX V->>DB: visit.status = COMPLETED V->>DB: INSERT NotificationSaga (status=PENDING) V->>MQ: publish visit.completed V->>DB: COMMIT MQ->>M: deliver visit.completed M->>SMTP: send mail SMTP-->>M: 250 OK M->>MQ: publish visit.notification.ack MQ->>V: deliver notification.ack V->>DB: UPDATE saga SET status=COMPLETED

4.2. Failure path — mail fail vĩnh viễn

sequenceDiagram autonumber participant V as visits-service participant DB as visits.DB participant MQ as RabbitMQ participant M as mailer-service (Go) participant SMTP as Mailpit (SMTP) participant Admin as admin alert consumer V->>DB: visit.status = COMPLETED + saga PENDING V->>MQ: publish visit.completed MQ->>M: deliver M->>SMTP: send mail SMTP-->>M: 550 mailbox not found M->>MQ: publish visit.notification.failed MQ->>V: deliver notification.failed V->>DB: UPDATE saga SET status=COMPENSATED V->>MQ: publish visit.manual-followup-required MQ->>Admin: alert (ops/vet liên hệ tay)

4.3. State machine của NotificationSaga

PENDING ──ack───────▶ COMPLETED (happy path, terminal) │ └──ack-failed────▶ COMPENSATED (compensation triggered, terminal)

Không có CANCELLED — saga không bị cancel giữa chừng. Mọi saga đều resolve qua 1 trong 2 đường.

5. Implementation chi tiết

5.1. Shared module — generic envelopes

shared/common-events/src/main/java/.../saga/NotificationAck.java
public record NotificationAck(
    UUID eventId, String eventType, Instant occurredAt, String source,
    UUID originalEventId,    // correlation tới event gốc
    String domain,           // "visit" | "user" | "invoice"
    String entityId,         // domain entity ID (string để generic)
    String recipient
) {
    public String routingKey() { return domain + ".notification.ack"; }
}

Lý do generic: 1 envelope dùng cho mọi saga notification (visit, user, invoice...). Mailer publish 1 type, mọi service initiator bind queue với routing pattern domain của mình.

5.2. Initiator — visits-service

visits-service/.../service/impl/VisitServiceImpl.java
@Transactional
public VisitResponse complete(Long id, CompleteVisitRequest req) {
    Visit v = loadOrThrow(id);
    v.complete(req.diagnosis(), req.treatment(), req.fee());   // status = COMPLETED
    publishCompleted(v);                                        // saga start + event publish
    return VisitResponse.from(v);
}

private void publishCompleted(Visit v) {
    // ... build VisitCompletedEvent ...
    sagaRepo.save(NotificationSaga.start(
        event.eventId(), v.getId(), "visit.completed.notification"));
    publisher.publish(event);
    // ⚠️ Dual-write: nếu broker fail sau DB commit → saga PENDING orphan.
    //    Prod cần Transactional Outbox (xem §7).
}

5.3. Notifier — Go mailer publish ack/failed

mailer-service/cmd/mailer/main.go
sendErr := m.Send(ctx, ev.CustomerEmail, ...)
publishSagaAck(ctx, cons, "visit", strconv.FormatInt(ev.VisitID, 10),
    ev.EventID, ev.CustomerEmail, sendErr, log)
// trả nil dù mail fail — saga event đã publish, không double-signal qua DLQ
return nil

// publishSagaAck quyết định ack vs failed dựa trên sendErr
func publishSagaAck(...) {
    if sendErr == nil {
        cons.Publish(ctx, "visit.notification.ack", NotificationAck{...})
    } else {
        cons.Publish(ctx, "visit.notification.failed", NotificationFailed{
            ...,
            ErrorMessage: sendErr.Error(),
        })
    }
}

5.4. Saga handler — compensating transaction

visits-service/.../saga/NotificationSagaHandler.java
@RabbitListener(queues = "visits.visit.notification.ack")
@Transactional
public void onNotificationAck(NotificationAck payload) {
    NotificationSaga saga = sagaRepo.findByEventId(payload.originalEventId()).orElseThrow(...);
    if (saga.getStatus() != SagaStatus.PENDING) return;   // idempotent
    saga.markCompleted();
}

@RabbitListener(queues = "visits.visit.notification.failed")
@Transactional
public void onNotificationFailed(NotificationFailed payload) {
    NotificationSaga saga = sagaRepo.findByEventId(payload.originalEventId()).orElseThrow(...);
    if (saga.getStatus() != SagaStatus.PENDING) return;
    saga.markCompensated(payload.errorMessage());

    // Compensating action: business-level, NOT DB rollback
    publisher.publish(VisitManualFollowUpRequiredEvent.of(
        visitId, payload.originalEventId(), payload.errorMessage(),
        customerId, username, email));
}

Compensation = business action. Visit đã COMPLETED (vet đã khám xong) — KHÔNG thể "uncomplete". Compensation là mitigation: alert ops liên hệ khách hàng thủ công. Đây là pattern phổ biến cho non-financial saga.

6.1. Transactional Outbox (chưa làm — TODO prod)

Hiện tại publishCompleted() save saga row + publish event trong cùng @Transactional. Tuy nhiên:

  • DB commit OK + broker publish FAIL → saga PENDING orphan
  • DB rollback + broker publish OK → mailer ack saga không tồn tại

Fix: Bảng outbox_events trong cùng DB. Service ghi event vào outbox trong TX (atomic). Background poller (Spring @Scheduled hoặc Debezium CDC) đọc outbox, publish broker, đánh dấu published_at.

CREATE TABLE outbox_events (
  id BIGSERIAL PRIMARY KEY,
  aggregate_id VARCHAR(64) NOT NULL,
  event_type VARCHAR(80) NOT NULL,
  routing_key VARCHAR(120) NOT NULL,
  payload JSONB NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  published_at TIMESTAMPTZ
);
CREATE INDEX idx_outbox_unpublished ON outbox_events (created_at) WHERE published_at IS NULL;
