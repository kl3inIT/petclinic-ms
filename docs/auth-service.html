auth-service

Service xác thực và phát hành JWT cho toàn cluster — quản lý User, role, refresh token rotation, và JWKS endpoint cho service khác validate token. Identity provider của hệ thống.

RS256 JWT signing với keypair generated tại startup, public key expose qua /.well-known/jwks.json. Access token 15 phút, refresh token 7 ngày — refresh single-use rotation với detect-reuse → revoke all (signal token theft). User identity = UUID (không expose Long internal). Roles = CSV trong DB (đơn giản hơn N-N table). Phát event user.registered cho mailer.

2. Personas & roles

👤 USER (default sau register)

Tự đăng ký, login, refresh token, đổi mật khẩu, xem profile của mình. Mọi service downstream check role này cho ownership endpoints.

🛡️ STAFF

Receptionist — quyền tạo owner hộ, book visit hộ, moderation review. Role này gán thủ công bởi admin (chưa có UI gán role — phase next).

🩺 VET

Bác sĩ — quyền start + complete visit, set ca trực (phase next). Gán bởi admin khi tạo account.

👑 ADMIN

Toàn quyền + gán role + hard delete + catalog admin (specialties, pet types). Tạo qua seed Liquibase (không qua register endpoint).

Role được lưu CSV string (vd. roles_csv = "USER,STAFF") thay vì entity Role + N-N table. Lý do: roles tĩnh (4 giá trị, không thêm runtime), CSV đủ + zero JOIN overhead. Trade-off: query "tìm user có role X" phải dùng LIKE '%X%' — chấp nhận vì rare. Nếu sau này roles động (hundreds of tenants) → migrate sang N-N.

Vì sao RS256 thay vì HS256: HS256 dùng shared secret — mọi validator cần biết secret = secret leak ở 1 service = compromise toàn cluster. RS256 asymmetric: chỉ auth-service biết private key (sign), mọi service khác chỉ cần public key (verify). Public key đăng qua JWKS endpoint, validator tự fetch + cache. Đúng zero-trust principle.

4. ERD — User + RefreshToken

erDiagram USER ||--o{ REFRESH_TOKEN : "có nhiều (1 per device)" USER { uuid id PK "GenerationType.UUID" string username "UNIQUE NOT NULL length 100" string email "NOT NULL length 100" string password "BCrypt hash 12 rounds" string roles_csv "Set qua StringSetConverter" bool enabled "default true — disable thay vì delete" timestamp created_date timestamp last_modified_date string created_by string last_modified_by } REFRESH_TOKEN { uuid id PK uuid user_id FK "INDEX" string token_hash "SHA-256 hex, UNIQUE INDEX" timestamp expires_at "now + 7 days" bool revoked "default false" timestamp created_date }

Quyết định thiết kế quan trọng

  • User.id là UUID, không Long: ID xuất hiện trong JWT sub claim → expose ra public. UUID không leak insertion order, không enumerable.
  • Password = BCrypt 12 rounds: ~250ms/hash trên dev machine — chống brute force. KHÔNG dùng SHA hay MD5.
  • RefreshToken lưu hash, không raw: DB leak ≠ token usable. Verify: hash input token + lookup. Tốc độ SHA-256 OK vì refresh endpoint không hot.
  • Raw refresh token = random 256-bit hex (32 bytes từ SecureRandom), KHÔNG phải UUID — UUID có structural bits, không đủ entropy cho secret.
  • Single-use rotation: mỗi refresh tạo token mới + mark old revoked=true. Nếu token đã revoked được dùng lại → detect theft → revoke ALL của user (Sequence 4).
  • UNIQUE INDEX token_hash: phòng race condition khi 2 device cùng refresh đồng thời (chỉ 1 thắng).

sequenceDiagram autonumber actor V as Victim (legit user) actor T as Thief (token stolen) participant A as auth-service participant RTS as RefreshTokenService participant DB as Postgres Note over V,T: Thief đã chiếm được refresh token R1 (vd. → chỉ có thể là thief RTS->>DB: UPDATE refresh_token SET revoked=true
WHERE user_id=victimId (ALL TOKENS) RTS-->>A: throw InvalidRefreshTokenException A-->>T: 401 Note over A: AuthAuditLogger.securityIncident
("Refresh token reuse detected") Note over V: Victim's R2 cũng bị revoke → next request 401 → force re-login.
Đau cho UX nhưng đúng — không biết R2 có bị leak chung không.
Trade-off của reuse detection: victim bị force logout cùng thief — không phân biệt được session nào "lành". Đây là cost of defense: thà invalidate tất cả còn hơn để thief tiếp tục access. Phase next có thể thêm device fingerprint trên refresh_token để chỉ revoke device bị nghi.

Defense-in-depth (gotcha #11): JWT được validate ở CẢ gateway VÀ downstream service. Không phụ thuộc gateway forward header X-User-Id — luôn re-verify Bearer token. Nếu attacker bypass gateway (vd. internal misconfiguration), downstream vẫn từ chối invalid token.

13. Câu hỏi mở

  1. Email verification: Bắt buộc verify mới login được? Hay cho login + nhắc verify? Tác động đến UX onboarding.
  2. Role bootstrap: Sao tạo user đầu tiên với role ADMIN? Hiện chỉ qua Liquibase seed — phase next có thể thêm CLI command hoặc env var bootstrap.
  3. Token storage FE: Hiện localStorage (vulnerable XSS) hay httpOnly cookie (cần CSRF protection)? Trade-off.
  4. Logout cascade: Revoke all refresh tokens của user — có nên cũng push event user.logged-out để invalidate cache ở các service khác (vd. Spring Boot Admin)?
  5. Session timeout: Inactive user > X giờ → force logout? Cần thêm last_seen_at trên User?
  6. Password policy: Hiện chỉ check length ≥ 8. Thêm complexity (uppercase + number + special)? Có nên check qua HaveIBeenPwned API?
  7. Anti-enumeration: Hiện trả 401 cùng message cho wrong username và wrong password — tốt. Nhưng register vẫn trả 409 cho username taken → vẫn enumerable. Có nên unified message "đã có tài khoản dùng email/username này"?
  8. Prod key rotation timeline: Bao lâu rotate 1 lần? 90 ngày? 1 năm?
