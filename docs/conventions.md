# Conventions

## Java And Spring

- No Lombok.
- DTOs are Java records. JPA entities are plain classes.
- Controllers depend on service interfaces. Implementations live under
  `service/impl`.
- Conversions live on DTO records as `from(...)` and `toEntity()` helpers. Do
  not add MapStruct or ModelMapper.
- Service packages follow:

```text
com.mss301.petclinic.<service>/
  config/
  controller/
  service/
    impl/
  repository/
  model/
  dto/
    req/
    res/
  exception/
```

- API paths use `/api/v1/...`. Do not break v1; add v2 alongside when needed.
- Controller method names must be unique across services because Spring uses
  them as OpenAPI `operationId` values and the gateway aggregates specs.
- Domain-specific exceptions extend shared exceptions from `common-web`.
  Services do not define their own `ExceptionTranslator`.

## Configuration

- Use `@ConfigurationProperties` records for related config keys.
- Split property records from activator configuration:
  `FooProperties.java` plus `FooPropertiesConfiguration.java`.
- Use `@DefaultValue` on record parameters for visible defaults.
- Use `@Value` only for one-off keys read by one bean.
- Add the Spring Boot configuration processor when a service defines new
  properties.

## Persistence

- PostgreSQL + JPA + Liquibase is the default.
- Hibernate uses `ddl-auto: validate`.
- Liquibase changesets create the service schema first, then use explicit
  schema names.
- Liquibase tracking tables stay in `public`.
- Entities that extend `AbstractAuditingEntity` need matching auditing columns
  in Liquibase.
- Enums persisted to the database implement `IdentifiedEnum` or `OrderedEnum`;
  persist stable IDs, not raw `Enum.name()`.

## Frontend

- API clients and DTOs are generated by orval. Do not edit generated files.
- Forms use `@tanstack/react-form` with Zod. Do not reintroduce
  `react-hook-form`.
- API helper code that cannot be generated lives next to the feature.
- Preserve `apiClient.baseURL = ""` because generated functions emit full
  `/api/v1/...` paths.

## Git And Verification

- Preserve unrelated user changes in the working tree.
- Do not run `gradle clean` unless explicitly requested.
- Prefer focused verification for the touched module, then broader checks when
  shared contracts change.
