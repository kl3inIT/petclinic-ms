customers-service

Service quản lý chủ thú (Owner) và thú cưng (Pet) — service đầu tiên của project (Phase 1, smallest surface) và là foundation aggregate cho mọi flow phía customer.

Owner 1-N Pet (unidirectional — Pet không back-ref Owner). Pet aggregate root vẫn là Owner: mọi write Pet đi qua endpoint /owners/{id}/pets. GET /pets/{id} đứng riêng chỉ để cross-service (visits-service) đọc và validate ownership. Pet.type là String đơn giản (catalog tự do), không phải entity PetType như Champlain — giảm scope, tránh master-data overhead.

2. Personas & quyền

👤 Customer (USER)

Read: profile của chính mình (Owner + pets).

Update: tự sửa profile + thêm pet mới (phase next).

Không thể: xem owner khác.

🛡️ Staff (RECEPTIONIST)

Toàn quyền trên owner + pet — tạo owner hộ khách (khi gọi điện đăng ký), sửa thông tin, thêm pet.

👑 Admin

Quyền Staff + delete (cascade xóa toàn bộ pet). Cẩn thận: xóa owner → mọi visit cũ tham chiếu pet_id sẽ orphan ID.

3. Bounded context

Sở hữuKhông sở hữu
  • Owner identity profile (tên, địa chỉ, SĐT)
  • Pet identity (tên, loài, ngày sinh)
  • Quan hệ Owner ↔ Pet
  • Visit lịch sử — visits-service sở hữu
  • Login credential — auth-service sở hữu (Owner ≠ User)
  • Photo binary — files-service sở hữu (phase tương lai)
  • Bill — billing-service sở hữu
Owner ≠ User — Owner là domain entity (hồ sơ khách hàng), User là identity (account login). Hiện tại không link — staff có thể tạo Owner cho khách chưa có account. Phase next sẽ thêm optional owner.user_id để self-service.

4. ERD — Owner aggregate

erDiagram OWNER ||--o{ PET : "có nhiều" OWNER { bigint id PK string first_name "NOT NULL" string last_name "NOT NULL" string address string city string telephone uuid user_id "FUTURE — link auth-service" timestamp created_date timestamp last_modified_date string created_by string last_modified_by } PET { bigint id PK bigint owner_id FK "unidirectional — Pet không có entity ref ngược" string name date birth_date string type "String tự do (dog/cat/bird...)" timestamp created_date timestamp last_modified_date }

Ràng buộc & quyết định thiết kế

  • Unidirectional 1-N: Pet entity có field owner_id nhưng insertable=false, updatable=false — chỉ mirror đọc. Hibernate quản FK qua Owner.@OneToMany @JoinColumn(name="owner_id"). → JSON serialize Pet đơn giản (không có Owner lồng).
  • Cascade ALL + orphan removal: xóa Owner → cascade xóa toàn bộ Pet. Remove pet khỏi owner.pets list → Hibernate auto DELETE Pet (orphan).
  • Pet.type là String: Champlain có entity PetType riêng (catalog dog/cat/bird). Petclinic-ms đơn giản hóa thành String — đủ cho MVP, validate ở DTO level. Khi cần i18n hoặc filter advanced → upgrade thành enum implementing IdentifiedEnum.
  • INDEX (last_name) cho filter ?lastName= trong list.

Vì sao PetController đứng riêng: Pet là child của Owner aggregate (write qua /owners), nhưng visits-service cần endpoint GET /pets/{id} để validate trước khi book. Nếu chỉ có GET /owners/{id}, visits phải tải nguyên owner + toàn bộ pet → wasteful. PetResponse còn trả ownerId để visits-service check ownership.

10. Câu hỏi mở

  1. Owner ↔ User link: 1-1 (1 user = 1 owner) hay 1-N (1 user quản nhiều owner, vd. con quản ba mẹ)? Đa số petclinic là 1-1 — đề xuất giữ đơn giản.
  2. Self-service registration: Customer tự đăng ký account ở auth-service → tự động tạo Owner trống? Hay phải đến clinic để staff tạo?
  3. PetType String vs entity: Khi nào upgrade? Trigger nếu cần filter ?type=cat nhiều, hoặc i18n label, hoặc gắn icon/avatar default per type.
  4. Soft delete owner: Owner có visit history → có nên cấm delete hoặc soft delete? Hiện hard delete cascade xóa pet — visit cũ vẫn tham chiếu pet_id nay không tồn tại.
  5. Telephone format: Validate VN format (+84xxx) hay tự do? Champlain dùng @Digits(integer=10) Bắc Mỹ — VN cần khác.
  6. Address structure: Hiện address + city 2 String. VN cần tỉnh/quận/phường/đường structured? Hay free text đủ?
