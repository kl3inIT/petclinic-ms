visits-service

Service quản lý vòng đời buổi khám — orchestrator kết nối customers (pet) + vets (bác sĩ) + auth (customer info). Chứa state machine SCHEDULED → IN_PROGRESS → COMPLETED + CANCELLED. Phát event cho mailer / billing / reviews tiêu thụ async.

Visit là aggregate trung tâm của clinic domain. Customer book → Vet start + complete (ghi diagnosis + treatment + fee) → publish visit.completed event → fan-out đến mailer (gửi mail) + billing (tạo invoice) + reviews (mở quyền review). Cross-service call qua RemoteClientsFacade + @CircuitBreaker (gotcha #28: bean tách riêng để Spring AOP intercept). Authorization: role ở filter chain + ownership ở service layer.

2. Personas & quyền

👤 Customer (USER)

Book: tạo visit cho pet của mình với vet bất kỳ vào slot trống.

Read: chỉ thấy visits của chính mình (filter customerUserId == JWT.sub).

Cancel: chỉ visit của mình, chỉ khi status = SCHEDULED.

Không thể: start, complete (đặc quyền của vet/staff).

🩺 Vet

Start: đánh dấu visit IN_PROGRESS khi khách đến.

Complete: đóng visit + ghi diagnosis, treatment, fee.

Read: thấy mọi visit (privileged role).

🛡️ Staff (RECEPTIONIST)

Book hộ: tạo visit thay customer (khách gọi điện đặt lịch).

Cancel: hủy bất kỳ visit nào (không bị giới hạn ownership).

Start: được phép (vd. lễ tân check-in khách trước khi vet vào phòng).

👑 Admin

Toàn quyền — bao gồm complete (xử lý trường hợp vet quên đóng visit).

4. ERD — Visit aggregate

erDiagram VISIT { bigint id PK bigint pet_id "ref customers-service (no FK)" bigint vet_id "ref vets-service (no FK)" uuid customer_user_id "ref auth-service (no FK)" timestamp scheduled_at "NOT NULL" string status "ENUM SCHEDULED IN_PROGRESS COMPLETED CANCELLED" text reason "lý do book" text diagnosis "fill khi complete" text treatment "fill khi complete" decimal fee "12,2 — fill khi complete" bigint version "@Version optimistic lock" timestamp created_date timestamp last_modified_date string created_by string last_modified_by }

Index & ràng buộc

  • UNIQUE (vet_id, scheduled_at) — chống double-book cùng vet cùng slot (đẩy lỗi qua SlotTakenException)
  • INDEX (customer_user_id, status) — query chính của customer: "visits của tôi đang diễn ra"
  • INDEX (vet_id, scheduled_at) — query lịch của vet trong ngày
  • INDEX (status, scheduled_at) — staff dashboard: visit SCHEDULED sắp tới
  • @Version — optimistic lock chống race condition khi 2 vet cùng cancel
Không có FK xuyên service — pet_id, vet_id, customer_user_id chỉ là ID logic. Validation xảy ra qua HTTP call lúc book (kiểm tra pet/vet tồn tại). Nếu downstream xóa pet → visit cũ vẫn còn (lịch sử), chỉ hiển thị "Pet đã bị xóa".

8. Sequence 3 — Cancel (USER ownership check ở service layer)

sequenceDiagram autonumber actor Actor as USER or STAFF participant GW as api-gateway participant V as visits-service (controller) participant Svc as VisitServiceImpl participant DB as Postgres Actor->>GW: PATCH /api/v1/visits/99/cancel GW->>V: forward + JWT V->>V: Filter chain: authenticated (cả USER + STAFF đều OK) V->>V: controller extract:
currentUserId = jwt.sub
isPrivileged = roles ∩ {STAFF,ADMIN,VET} != ∅ V->>Svc: cancel(99, currentUserId, isPrivileged) Svc->>DB: SELECT visit WHERE id=99 DB-->>Svc: Visit{customerUserId: abc-123, status:SCHEDULED} alt !isPrivileged && visit.customerUserId != currentUserId Svc->>Svc: throw AccessDeniedException("Không phải visit của bạn") Svc-->>V: 403 ProblemDetail else OK Svc->>Svc: visit.cancel() — guard: status in {SCHEDULED, IN_PROGRESS} alt status đã COMPLETED hoặc CANCELLED Svc--xV: IllegalVisitTransitionException → 400 ProblemDetail else OK Svc->>DB: UPDATE visit SET status='CANCELLED' Svc-->>V: VisitResponse V-->>Actor: 200 end end
Vì sao ownership ở SERVICE layer:
  • Nếu dùng @PreAuthorize("@securityHelper.isOwner(#id)") → cần load entity 2 lần (1 để check, 1 để cập nhật) → N+1 DB hit.
  • Ownership rule là domain logic, không phải security boundary → test domain service mà không phải khởi tạo Spring Security context.
  • Rule rõ ràng ở 1 chỗ duy nhất (không scatter giữa filter chain + annotations).

Gotcha #28 đã được áp dụng: @CircuitBreaker đặt ở RemoteClientsFacade bean RIÊNG (không trong VisitServiceImpl), tránh self-invocation bypass Spring AOP proxy. 14. Câu hỏi mở

  1. Slot length cố định? Hiện schema chỉ có scheduledAt, không có duration — mặc định 30 phút? Phẫu thuật cần 120 phút → cần thêm field hay đặt theo specialty?
  2. Multi-pet visit: 1 owner mang 2 pet đến cùng lúc → 2 visit riêng hay 1 visit với List<Pet>? Hiện thiết kế 1 visit = 1 pet.
  3. No-show: Customer không đến — staff đánh dấu thế nào? Thêm status NO_SHOW hay dùng CANCELLED với reason?
  4. Reschedule policy: Cho reschedule không giới hạn? Tối đa 2 lần? Trước 24h?
  5. Vet deactivation: Khi vet nghỉ việc (vets-service publish vet.deactivated?) → auto-cancel visit SCHEDULED hay notify staff để reassign?
  6. Fee currency: Hiện BigDecimal không có currency. Multi-currency không? Hay cố định VND?
  7. Receipt: Sau complete, customer nhận PDF biên lai (files-service generate)? Hay chỉ email plain text?
