petclinic-ms — Business Flow

Hệ thống quản lý phòng khám thú y kết hợp pet shop. Microservices Spring Boot 4 + React 19. Tài liệu mô tả nghiệp vụ, personas, workflows và bounded contexts.

Số hóa toàn bộ chu trình: khách hàng đăng ký pet → đặt lịch khám → bác sĩ chẩn đoán → thanh toán → mua phụ kiện. 4 personas (Customer / Vet / Staff / Admin), 3 domain (Clinic / Shop / Platform), 10 services. Phân tách bounded context theo team ownership + deploy cadence + compliance scope.

1. Tầm nhìn hệ thống

Bản gốc Spring Petclinic chỉ là phòng khám đơn thuần — owner đăng ký pet, đặt visit, vet ghi diagnosis. petclinic-ms (MSS301) mở rộng thành full clinic + e-commerce hybrid:

2. Personas — 4 vai chính

🐱 Customer (Pet Owner)

Role: USER

Cổng: Public + Customer area

Hành vi: Self-service

Đăng ký tài khoản, thêm pet của mình, đặt lịch khám, xem hóa đơn, mua phụ kiện/thức ăn trong shop.

🩺 Vet (Bác sĩ)

Role: VET

Cổng: Admin (role-gated)

Hành vi: Thao tác chuyên môn

Xem lịch khám của mình, xem hồ sơ pet, ghi diagnosis + đơn thuốc, đóng visit khi xong.

👤 Staff (Receptionist)

Role: STAFF

Cổng: Admin (full operations)

Hành vi: CRUD operations

Tiếp tân walk-in, tạo owner+pet, schedule visit, tạo bill, xác nhận thanh toán, manage inventory.

⚙️ Admin

Role: ADMIN

Cổng: Admin (full + config)

Hành vi: System governance

Quản lý user/role, xem dashboard doanh thu + KPI, đọc audit log, cấu hình system, manage products catalog.

4. Service inventory — chi tiết trách nhiệm

ServiceAggregate / Domain objectTrách nhiệm businessTại sao tách
auth-service User, Role, RefreshToken, AuditEvent Đăng ký, đăng nhập, cấp JWT RS256, refresh token rotation, audit login event, publish JWKS Cô lập compliance (lưu credential), scale-out riêng khi traffic register peak
customers-service Owner ↔ Pet (1-N) CRUD owner, pet; mỗi pet thuộc 1 owner; lịch sử pet (sinh nhật, giống, cân nặng) Aggregate đóng kín: 1 owner sở hữu nhiều pet, không invariant cross-service
vets-service Vet ↔ Specialty (N-N) Danh sách bác sĩ + chuyên môn (mèo, chó, thú lạ, phẫu thuật...) Đội ngũ tách biệt customer, quản lý ca trực, chuyên môn — domain khác hẳn
visits-service next Visit (PetId, VetId, ScheduledAt, Status, Diagnosis) Đặt lịch (kiểm tra conflict slot), xác nhận, ghi diagnosis, đóng visit, lịch sử khám Lịch là pain point: race condition booking, reminder cron, history dài → schema + index khác hẳn
billing-service todo Invoice, Payment, LineItem Tạo hóa đơn từ visit fee + product order, xử lý thanh toán, refund Idempotency cực kỳ quan trọng + PCI-DSS scope + integrate Stripe/VNPay sau
products-service todo Product, Category, Price Catalog: thức ăn, đồ chơi, thuốc OTC, phụ kiện. Search + filter + featured Read-heavy, cacheable, CDN-friendly — pattern khác hẳn write-heavy clinic services
cart-service todo Cart (UserId, Items, TTL) Giỏ hàng theo user, có TTL 24h, checkout → trigger billing + inventory Stateful per-user, ephemeral — hợp Redis-backed thay vì Postgres
inventory-service todo Stock, StockMovement Quản lý số lượng tồn, nhập/xuất, trừ kho khi checkout, cảnh báo low-stock Concurrent update cần atomic (SELECT ... FOR UPDATE); business exception "không đủ hàng"
mailer-service todo EmailEvent, Template Gửi email: xác nhận đăng ký, nhắc visit, biên lai, reset password Async via queue (Kafka/RabbitMQ), retry, không block luồng business chính
files-service todo FileObject (key, contentType, size, ownerId) Upload ảnh pet, document hồ sơ y tế, presigned URL download Binary lớn — không hợp JPA + Postgres, tách để dùng MinIO/S3
api-gateway — Routing lb://, validate JWT, CORS, per-IP rate limit, circuit breaker, fallback Single entry point — đóng port cho tất cả service backend

9. Business rules quan trọng

RuleService enforceCơ chế
Owner chỉ thấy pet của mình customers Query filter WHERE owner_id = jwt.sub; method-level @PreAuthorize
Visit không double-booking 1 vet visits Unique constraint (vet_id, scheduled_at) + optimistic locking
Vet chỉ xem được visit của mình visits Query filter theo JWT sub + role VET
Stock không bao giờ âm inventory SELECT FOR UPDATE + check quantity >= requested → exception nếu fail
Refund tự động khi cancel visit < 24h trước slot visits → billing Domain event "VisitCancelled" với timestamp → billing tính refund policy
Bill phải PAID trước khi đóng visit visits + billing State machine ở visits, check billing.status qua REST hoặc cached projection
Audit mọi thao tác đụng tiền + JWT login auth + billing Append-only audit table (ECS-compliant log fields)
Rate limit register/login để chống brute-force gateway Per-IP token bucket (10 req/min cho auth endpoints)
Refresh token reuse → revoke all tokens user đó auth Detect reuse (token đã rotated nhưng client còn dùng) → security incident

10. Tại sao tách microservices (business view)

Lý doVí dụ trong petclinic-ms
Team ownership khác nhau Clinic team sở hữu customers/vets/visits. Shop team sở hữu products/cart/inventory. Platform team sở hữu auth/gateway/files.
Tần suất deploy khác nhau Products catalog đổi hàng ngày (marketing campaign). auth-service đổi 1 lần/quý (vì compliance review).
Scale profile khác nhau products GET cực cao → CDN + cache. billing POST hiếm hơn nhưng critical → HA cluster.
Compliance scope khác nhau billing trong PCI-DSS scope (xử lý card data → thuê auditor). Medical records cô lập (HIPAA-like). Files cô lập (S3 access policy).
Tech stack flexibility files dùng MinIO + presigned URL. mailer dùng worker queue. cart dùng Redis. Không ép tất cả là Postgres+JPA.
Failure isolation mailer chết không kéo theo clinic chết. inventory bug không làm sập auth.
Data ownership rõ ràng Mỗi service 1 schema Postgres riêng. Không share DB → không phụ thuộc schema chéo. Cross-service đọc qua REST hoặc event.
