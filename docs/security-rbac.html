Mỗi Spring service khai báo endpoint rules trong config-repo/<service>.yml dưới prefix petclinic.security.endpoints.* với 5 buckets: public, user, staff, admin, customRoles (vd VET). EndpointSecurityCustomizer (servlet) hoặc ReactiveEndpointSecurityCustomizer (WebFlux) đọc properties + áp rules vào SecurityFilterChain. Visits-service thêm layer 2: @PreAuthorize("@visitSecurity.canView(#id, authentication)") cho per-instance ownership.

1. Concepts

Role hierarchy

ADMIN > STAFF > USER (implicit qua hasAnyRole listing trong customizer) ADMIN > VET (custom role, áp riêng cho visits) PUBLIC (no auth needed)

7. How the customizer works

Servlet flow

shared/common-security/.../endpoints/EndpointSecurityCustomizer.java
public static void apply(
        AuthorizationManagerRequestMatcherRegistry registry,
        SecurityEndpointsProperties props) {
    for (Endpoint e : props.publicEndpoints())  applyPermitAll(registry, e);
    for (Endpoint e : props.adminEndpoints())   applyRole(registry, e, "ADMIN");
    props.customRoles().forEach((role, endpoints) -> { ... });   // VET, etc.
    for (Endpoint e : props.staffEndpoints())   applyAnyRole(registry, e, "STAFF", "ADMIN");
    for (Endpoint e : props.userEndpoints())    applyAnyRole(registry, e, "USER", "STAFF", "ADMIN");
}

Order matters

Spring Security match request theo thứ tự khai báo, dừng ở match đầu tiên. Customizer áp theo thứ tự specific → broad:

  1. Public — permitAll, skip role check
  2. Admin — narrowest role
  3. Custom roles (VET, etc.)
  4. Staff
  5. User — broadest role, áp cuối

Lý do: nếu áp USER trước, một request match USER pattern sẽ stop, không bao giờ đến rule ADMIN-narrower phía sau.

8. Per-instance authorization

URL coarse-grained KHÔNG biểu đạt được "USER chỉ xem visit của chính mình". Path /api/v1/visits/123 chỉ biết user authenticated, không biết visit 123 thuộc ai. Cần load entity → so sánh ownership.

visits-service/.../security/VisitSecurity.java
@Component("visitSecurity")
public class VisitSecurity {
    private static final Set<String> PRIVILEGED_ROLES =
            Set.of("ROLE_STAFF", "ROLE_ADMIN", "ROLE_VET");

    public boolean canView(Long visitId, Authentication auth) {
        if (visitId == null || !auth.isAuthenticated()) return false;
        if (hasPrivilegedRole(auth)) return true;          // STAFF/ADMIN/VET bypass
        return isOwnerOf(visitId, auth);                    // USER → load + compare
    }

    private boolean isOwnerOf(Long visitId, Authentication auth) {
        UUID currentUserId = extractUserId(auth);
        return visitRepo.findById(visitId)
                .map(Visit::getCustomerUserId)
                .map(currentUserId::equals)
                .orElse(false);
    }
}

Annotation gọi bean theo tên qua SpEL:

@PreAuthorize("@visitSecurity.canView(#id, authentication)")

Tại sao bean tách riêng?

  • Reusable: mọi controller method (view, cancel, update...) dùng cùng bean
  • Unit-testable: mock VisitRepository, assert decision không cần boot HTTP
  • SpEL clean: annotation 1 dòng thay vì block code if/else trong controller
